The separation is deliberate: first decide what deserves release, then prepare the needed video material, then organize the final publishing materials. These stages exchange context, but each has a different definition of “done.”

## Gate 1: decide the release candidate

The first gate is editorial. The operator decides which track and version are worth finishing. Review notes should explain the choice well enough that the project does not restart from zero the next day.

Beginning video work before this gate is stable creates expensive ambiguity. A visual concept may attach itself to a track that is later rejected, or a new audio version may invalidate timing and edit decisions. A clear candidate state does not eliminate revision, but it makes the risk visible.

## Gate 2: prepare the visual materials

Video work has its own inputs and decisions: aspect ratios, source clips, edit versions, captions, cover frames, and export targets. These should be connected to the approved release record without being mistaken for the audio decision itself.

A useful handoff answers:

- which audio version the visual work uses;
- which visual deliverables are required;
- what is draft, approved, or blocked;
- where the local source and export files live;
- which decision still requires human review.

This record is more useful than a single “video done” checkbox because it tells the next operator—or the same operator a week later—what “done” actually means.

## Gate 3: assemble the publishing package

Publishing preparation is another distinct gate. It may include titles, descriptions, credits, cover assets, release dates, platform-specific copy, and distribution status. The materials can be prepared in advance, but publication should remain an explicit approval.

This separation prevents two common mistakes: assuming that a finished video means the release package is complete, and assuming that complete metadata means the creative work has been approved.

## Keep one handoff record across the gates

Separate stages do not require isolated silos. The useful connection is a small shared record that identifies the release candidate, current gate, next action, relevant local files, and approval status. Each specialist tool can continue doing the job it does best.

If a project stalls, the record should reveal whether the block is:

1. the song decision;
2. an audio or master requirement;
3. a missing visual asset;
4. unresolved publishing information; or
5. final operator approval.

That clarity is the reason to separate the work. The goal is not to add ceremony. It is to avoid restarting downstream tasks, losing context between tools, and confusing “prepared” with “approved.”
